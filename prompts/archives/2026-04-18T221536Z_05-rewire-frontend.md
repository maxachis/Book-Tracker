# Rewire Frontend to Wails Bindings

## Goal

Swap every `invoke("…")` call in the frontend for the generated Wails binding, replace all direct SQL via `@tauri-apps/plugin-sql` with calls to the Go `App` methods added in prompts 02–04, and delete the `@tauri-apps/*` dependencies entirely.

After this prompt, the app should run end-to-end under `wails dev`.

## Context

The Wails CLI generates `wails/frontend/wailsjs/go/main/App.{js,d.ts}` with one export per exported `App` method. Calls look like:

```ts
import { CreateBook, ListActiveBooks, ParseCSVBooks } from "../../wailsjs/go/main/App";
const book = await CreateBook(request);
```

## File-by-file changes

### `wails/frontend/src/lib/services/database.ts`

- Remove `import Database from "@tauri-apps/plugin-sql";` and `import { invoke } from "@tauri-apps/api/core";`.
- Replace `getDb()` and every inline SQL call with calls to the generated `App` methods:
  - `createBook` → `CreateBook(request)` (Go returns the fully populated `Book`).
  - `getActiveBooks` → `ListActiveBooks()`.
  - `getCompletedBooks` → `ListCompletedBooks()`.
  - `getAllBooks` → `ListAllBooks()`.
  - `updateBook` → `UpdateBook(request)`.
  - `deleteBook` → `DeleteBook(id)`.
  - `markBookComplete` → `MarkBookComplete(id)`.
  - `getSettings` → `GetSettings()`.
  - `updateSettings` → `UpdateSettings(request)`.
- The validation calls previously done client-side via `invoke("validate_…")` now happen inside the Go CRUD methods. Drop the redundant pre-validation `invoke` calls unless a specific view needs to validate before mutating — if it does, use `ValidateCreateBookRequest` / `ValidateProgressUpdate` / `ValidateSettings` (which return `""` on success, error string otherwise).

### `wails/frontend/src/lib/services/csv.ts`

- Remove `import { invoke } from "@tauri-apps/api/core";`.
- Replace `invoke("parse_csv_books", …)` with `ParseCSVBooks(contents)`.
- Replace `invoke("check_duplicates", …)` with `CheckDuplicates(records)` (the Go wrapper reads existing books itself).
- Replace `invoke("generate_csv_export", …)` with `GenerateCSVExport(books)`.
- Combine with `OpenCSVFile` / `SaveCSVFile` from prompt 04 for the dialog + I/O flow.

### Views / components

- `grep -rn "invoke\|@tauri-apps" wails/frontend/src` should return zero matches after this prompt.
- Remove any now-dead imports.

### Types

- If generated bindings produce slightly different shapes (e.g. `null` vs `undefined` on optional fields), adjust TS types in `wails/frontend/src/lib/types/index.ts` to match what Go returns. Prefer aligning TS to Go rather than wrapping values.

### Dependencies

- Remove from `wails/frontend/package.json`:
  - `@tauri-apps/api`
  - `@tauri-apps/plugin-sql`
  - (plus any leftover `@tauri-apps/*` from earlier prompts)
- Run `npm install` (or `npm prune`) and verify `package-lock.json` no longer lists Tauri packages.

## Guardrails

- **Do not** modify Go code in this prompt unless a binding mismatch surfaces — in that case, fix the Go signature (it is the source of truth) and note it in the commit.
- **Do not** add a re-export shim that keeps the `invoke(...)` call shape. The whole point is to let TypeScript type-check against the generated bindings.
- Keep the reactive state classes in `stores/state.svelte.ts` as-is structurally; only swap the service calls they depend on.
- If a `Book` field was serialized as `null` from Rust but `undefined` in TS, the Go side emits `null` for pointer/nullable fields — accept that and update the TS type to `string | null`.

## Smoke checks

- `npm run check` (svelte-check / tsc) passes with no `@tauri-apps` module-resolution errors.
- `wails dev` launches, lists books from an existing DB, lets you add/edit/complete/delete a book, and import/export CSV.
- `grep -rn "@tauri-apps\|invoke(" wails/frontend/src` returns nothing.
